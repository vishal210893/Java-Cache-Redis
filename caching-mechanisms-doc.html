Caching is the single highest-leverage performance optimization in distributed systems. Every major system design interview — from URL Shortener to Newsfeed to Chat — eventually asks: "How would you add caching?" The differentiator is knowing which pattern to use, why, and what breaks when you choose wrong.

This document covers six core read/write strategies, four eviction algorithms with worked examples, three invalidation approaches, three cache failure modes (penetration, breakdown, avalanche), and the distributed concerns (stampede, hot keys, consistency) that trip up most candidates. Companies that heavily test caching: Meta, Netflix, Amazon, Google, Twitter/X, Uber, Stripe, LinkedIn.

🎯 Interview Differentiator

Most candidates can name "write-through" and "write-back." The top 10% can explain when write-back is dangerous, derive the coalescing ratio that makes it worthwhile, and sketch the Lua script that prevents cache stampede. That's the depth this document targets.

Requirements

What a production caching layer must support — both functional behavior and non-functional targets.

Functional Requirements

Transparent Read Acceleration
Serve reads from cache when fresh data exists, falling back to source of truth on miss. The application should control staleness tolerance via TTL.
Configurable Write Strategy
Support write-through (synchronous consistency), write-back (async throughput), and write-around (skip cache on write) — selectable per data class.
Automatic Eviction
When cache memory is full, evict entries based on a configurable policy (LRU, LFU, or adaptive) without requiring application intervention.
Explicit Invalidation
Allow application code, CDC pipelines, or pub/sub events to invalidate specific keys or key patterns on data mutation.
Multi-Tier Topology
Support L1 (in-process) → L2 (distributed Redis) → origin DB hierarchy, with promotion/demotion between tiers.
Stampede Protection
Prevent thundering herd on cache miss — use probabilistic early expiry, locking, or request coalescing to limit origin load.

Data Model

The caching layer uses a tiered persistence strategy — fast volatile stores for the hot path, and a durable relational database as the source of truth.

🔴 Redis
L2 Distributed Cache
Primary distributed cache. Chosen over Memcached for: data structures (sorted sets, hashes), built-in TTL per key, Lua scripting for atomic cache-aside with locking, Redis Cluster with 16384 hash slots, and pub/sub for cross-node invalidation.
🟢 Memcached
Simple Key-Value Bulk Cache
High-volume, flat key→blob workloads. Multi-threaded architecture gives better per-node throughput for simple GET/SET. Slab allocator eliminates fragmentation. Ideal for sessions and page fragments.
⚡ Caffeine / Guava
L1 In-Process Cache
JVM-local cache with zero network overhead (~50ns). Caffeine uses Window-TinyLFU — near-optimal hit rate with O(1). Sized at 10–15% of heap. Reduces Redis round-trips by 50–80% for hot keys. Invalidated via Redis pub/sub.
🐘 PostgreSQL
Source of Truth
All strategies ultimately read from / write to PostgreSQL. Logical replication feeds CDC pipeline for event-driven invalidation. DB is authority — cache is always rebuildable. PgBouncer prevents cache-miss stampedes from exhausting connections.
🔑 Key Design Principle: Cache Is Derived State

The cache never holds data that doesn't exist in the source of truth (except during the write-back durability window). Every entry can be reconstructed from the DB. Cache failures are performance events, not data loss events.

Key Namespacing

Prefixing keys with {service}:{entity} prevents collisions across services sharing a Redis cluster, enables pattern-based invalidation (use SCAN with a glob match followed by pipelined DEL — Redis DEL itself does not support glob patterns), and makes memory profiling straightforward (redis-cli --bigkeys).

API Design

The cache abstraction layer exposes a clean interface that hides the chosen strategy from calling code.

Java — CacheService Interface
public interface CacheService<K, V> {

    Optional<V> get(K key);

    void put(K key, V value, Duration ttl);

    void invalidate(K key);

    // Bulk invalidation by prefix (SCAN + pipelined DEL)
    void invalidateByPrefix(String prefix);

    // Cache-aside helper: get-or-load atomically
    V getOrLoad(K key, Function<K, V> loader, Duration ttl);

    CacheStats stats();  // hits, misses, evictions, latency percentiles
}
Java — CacheClient Implementation
public class CacheClient<V> {
    private final Cache<String, V> l1;      // Caffeine
    private final RedisCommands<String, String> redis;  // Lettuce
    private final ObjectMapper mapper;

    public Optional<V> get(String key, Class<V> type) {
        // L1 check → L2 check → empty on miss
        V val = l1.getIfPresent(key);
        if (val != null) return Optional.of(val);

        String json = redis.get(key);
        if (json != null) {
            val = mapper.readValue(json, type);
            l1.put(key, val);  // promote to L1
            return Optional.of(val);
        }
        return Optional.empty();
    }

    public V getOrLoad(String key, Function<String, V> loader,
                       Class<V> type, Duration ttl) {
        return get(key, type).orElseGet(() -> {
            V val = loader.apply(key);
            put(key, val, ttl);
            return val;
        });
    }
}
⚠️ Why getOrLoad Matters

A naive get() → miss → load() → put() sequence has a race condition: 100 concurrent requests for the same missing key all issue DB queries simultaneously (a cache stampede). getOrLoad should internally acquire a per-key lock (or use singleflight/coalescing) so only one thread loads from DB while others wait.

Cache-Aside Pattern (Lazy Loading)

The most widely used caching strategy. Application code explicitly manages cache reads and writes — the cache is a "side" store, not in the data path.

Cache-Aside (Lazy Loading)

Application is responsible for cache reads/writes. On miss, app fetches from DB and populates cache. On write, app updates DB and invalidates cache. The cache library has no awareness of the DB.

Read/Write Flow

Implementation

Java — Cache-Aside with Jedis
private static final ThreadLocalRandom RNG = ThreadLocalRandom.current();

public User getUser(long userId) {
    String key = "user:profile:" + userId;

    // Step 1: Check cache
    String cached = jedis.get(key);
    if (cached != null) {
        if (cached.equals("NULL")) return null;  // negative cache hit
        return mapper.readValue(cached, User.class);
    }

    // Step 2: Cache MISS → fetch from DB
    User user = jdbcTemplate.queryForObject(
        "SELECT * FROM users WHERE id = ?", userRowMapper, userId);

    if (user == null) {
        jedis.setex(key, 60, "NULL");  // negative cache, short TTL
        return null;
    }

    // Step 3: Populate with TTL + jitter
    long ttl = 3600 + RNG.nextInt(300);
    jedis.setex(key, ttl, mapper.writeValueAsString(user));
    return user;
}

public void updateUser(long userId, UserUpdateRequest data) {
    // Step 1: Write to DB (source of truth FIRST)
    jdbcTemplate.update("UPDATE users SET name=?, email=? WHERE id=?",
        data.getName(), data.getEmail(), userId);

    // Step 2: Invalidate (NOT update — avoids race condition)
    jedis.del("user:profile:" + userId);
}
⚠️ Why Invalidate, Not Update?

On write, delete the cache key rather than update it. Two concurrent writes with SET can leave the cache permanently inconsistent with DB. Deleting ensures the next read fetches latest from DB. See also versioned keys for a stronger solution.

Negative Caching

Storing "NULL" for keys that don't exist in the DB. Without this, attackers flood your DB with non-existent IDs — every request is a miss. First line of defense against cache penetration. Short TTL (30–60s) so new records appear quickly. For stronger protection, pair with a Bloom filter.

Read-Through Cache

The cache library itself handles DB fetching on miss — app only talks to cache.

Read-Through Cache

Cache acts as the primary data interface. On miss, the cache (not the app) fetches from origin, stores the result, and returns it. Libraries: Caffeine LoadingCache, Hibernate L2, AWS DAX.

How It Works

The key difference from cache-aside: in cache-aside, your application code decides what to do on a miss (fetch from DB, serialize, store in cache). In read-through, the cache library itself has a loader function that knows how to fetch from the DB. You hand the cache a function at initialization time, and from that point on, your app code never touches the DB directly for reads — it only calls cache.get(key).

Built-in Stampede Protection — The Killer Feature

When 100 threads call cache.get("product:42") simultaneously and it's a miss, Caffeine's LoadingCache ensures only one thread calls the loader. The other 99 threads park on a CompletableFuture and receive the same result when the loader completes. This is singleflight/request coalescing built directly into the cache — no extra code, no distributed lock, no coordination.

Java — Caffeine LoadingCache with Async Refresh
// The loader: called ONCE per key per miss, even with 100 concurrent gets
LoadingCache<String, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofMinutes(15))
    .refreshAfterWrite(Duration.ofMinutes(12))  // refresh at 80% TTL
    .recordStats()  // expose hit rate, miss rate, eviction count
    .build(key -> userRepository.findById(key));  // loader

// Application code — this is ALL you write for reads:
User user = cache.get("user:12345");

Write-Through Cache

Every write goes through the cache layer synchronously to the database. Cache always consistent — at cost of write latency.

Write-Through

Every write updates both cache and DB synchronously. The app's cache strategy layer orchestrates both writes — first DB, then cache. Guarantees cache never holds data DB doesn't.

Implementation

Java — Write-Through
public class WriteThroughCache {
    private final JedisPool jedisPool;
    private final JdbcTemplate db;
    private final ObjectMapper mapper;

    public void put(String key, Object value, long ttlSeconds) {
        // Write to DB first (source of truth)
        db.update("INSERT INTO cache_store (key, val) VALUES (?, ?) "
            + "ON CONFLICT (key) DO UPDATE SET val = ?",
            key, mapper.writeValueAsString(value),
            mapper.writeValueAsString(value));

        // Then update cache (if DB write succeeds)
        try (Jedis j = jedisPool.getResource()) {
            j.setex(key, ttlSeconds, mapper.writeValueAsString(value));
        }
        // DB fail → exception, cache not updated (correct)
        // Cache fail → DB has data, cache stale → self-heals via TTL
    }
}

Consistency Model

🔑 Write Ordering: DB First, Then Cache

Always DB first. If process crashes between writes, worst case is stale cache — self-heals via TTL. Cache first risks cache claiming success while DB rejected the write.

✅ When to Use
  • Read-after-write consistency (profile updates)
  • Financial balances, inventory counts
  • Low write volume (<5% write ratio)
❌ When NOT to Use
  • High write throughput (counters, analytics)
  • Data written but rarely read
  • Write latency is critical

Write-Back (Write-Behind) Cache

Writes go to cache only, asynchronously flushed to DB. Optimizes write throughput at cost of durability risk.

Write-Back (Write-Behind)

Writes acknowledged as soon as they reach cache. Cache asynchronously flushes dirty entries to DB. Between cache write and flush, cache is the only copy — durability risk if cache node fails.

Implementation

Java — Write-Back with Atomic Dirty Set Swap
public class WriteBackCache {
    private static final String DIRTY = "cache:dirty_keys";
    private static final String PROCESSING = "cache:dirty_keys:processing";

    public void put(String key, String value) {
        // Atomic: write value + mark dirty
        try (Jedis j = pool.getResource()) {
            Pipeline p = j.pipelined();
            p.set(key, value);
            p.sadd(DIRTY, key);
            p.sync();
        }  // Caller returns immediately — DB write is async
    }

    /** Called by background ScheduledExecutor every N seconds */
    public void flushDirty() {
        try (Jedis j = pool.getResource()) {
            // RENAME atomically swaps dirty → processing.
            // New writes during flush go into fresh DIRTY set.
            // Avoids race where SMEMBERS+SREM drops mid-flush writes.
            try {
                j.rename(DIRTY, PROCESSING);
            } catch (JedisDataException e) {
                return;  // DIRTY doesn't exist → nothing to flush
            }

            // SSCAN for production scale (SMEMBERS blocks single-threaded Redis)
            Set<String> dirtyKeys = new HashSet<>();
            String cursor = "0";
            do {
                ScanResult<String> scan = j.sscan(PROCESSING, cursor,
                    new ScanParams().count(500));
                dirtyKeys.addAll(scan.getResult());
                cursor = scan.getCursor();
            } while (!cursor.equals("0"));

            if (dirtyKeys.isEmpty()) { j.del(PROCESSING); return; }

            // Fetch latest values (coalesced — only last write per key)
            List<String> keys = new ArrayList<>(dirtyKeys);
            List<String> vals = j.mget(keys.toArray(new String[0]));

            // Batch upsert — idempotent, so partial failure + retry is safe
            db.batchUpsert(keys, vals);

            j.del(PROCESSING);
        }
    }
}
💬 Question That Sparked This Section

"Why RENAME instead of SMEMBERS + SREM? Can you explain the race condition in more detail?"

Redis Set Commands — The Building Blocks

The write-back dirty tracker uses a Redis Set — an unordered collection of unique strings. Think of it as a HashSet<String> in Java, but stored in Redis so all application instances share it. Four commands are relevant here:

SADD — Add to Set

SADD dirty_keys "user:42" adds "user:42" to the set called dirty_keys. If the key is already in the set, SADD is a no-op (sets don't have duplicates). Returns 1 if added, 0 if already present. The application calls SADD every time it writes a cache value — marking that key as "dirty" (needs to be flushed to DB).

SMEMBERS — Get All Members

SMEMBERS dirty_keys returns every element in the set as a list: ["user:42", "product:99", "order:7"]. This is how the flush worker discovers which keys need to be written to the DB. Caveat: SMEMBERS returns all elements in a single blocking call — for large sets (100K+ keys), use SSCAN instead to iterate in batches without blocking Redis's single thread.

SREM — Remove from Set

SREM dirty_keys "user:42" "product:99" removes those specific elements from the set. The flush worker calls SREM after writing to the DB to say "these keys have been flushed, they're no longer dirty." This is where the race condition lives — if a key was re-dirtied between SMEMBERS and SREM, SREM still removes it.

RENAME — Atomically Rename a Key

RENAME dirty_keys processing_keys changes the key name from dirty_keys to processing_keys in a single atomic operation. After RENAME, dirty_keys no longer exists in Redis — any subsequent SADD dirty_keys ... from application threads creates a brand new, empty set. The old contents live under processing_keys where the flush worker can safely process them. This is the trick that gives us snapshot isolation.

SADD dirty_keys "order:7"           -- creates a NEW empty set, adds order:7
SMEMBERS processing_keys             -- → ["user:42", "product:99"] (frozen snapshot)
SMEMBERS dirty_keys                   -- → ["order:7"] (new writes, isolated)
🔑 Why Sets (Not Lists)?

If the app writes user:42 ten times between flushes, we only need to flush it once (with the latest value). A Redis Set automatically deduplicates — SADD on an existing member is a no-op. A List would store ten copies, causing the flush worker to write the same key to DB ten times. Sets give us free deduplication, which is the foundation of write coalescing.

The SMEMBERS + SREM Race Condition — Step by Step

The naive approach seems simple: read all dirty keys, fetch their values, write to DB, then remove them from the dirty set. But between reading the set and removing keys from it, the set is still live — application threads are adding new entries to it. This creates a window where data can be lost.

The Core Problem

SMEMBERS and SREM are two separate Redis commands. Between them, the dirty set can be modified by application writes. If a key is re-dirtied (written with a new value) after SMEMBERS but before SREM, the SREM removes it from the dirty set — even though the new value was never flushed. That key is now lost: the DB has the old value, Redis has the new value, and nothing will ever trigger a flush for it.

At t=2, the app wrote "v2" to Redis and added keyA back to the dirty set. But SADD on an already-present key is a no-op on the set membership — keyA was already there from t=0. After RENAME, the original dirty_keys key doesn't exist — any new SADD calls from application threads create a fresh, empty set. The flush worker processes the snapshot in the processing set while new writes accumulate in the new dirty set, completely isolated from each other.

t=6MGET(keyA) → "v2" ✅
DB.upsert(keyA="v2")		—∅{keyA}"v2" ✅
🔑 The Key Insight: Snapshot Isolation via RENAME

RENAME gives the flush worker a frozen snapshot of the dirty set at a single point in time. The worker processes this snapshot at its own pace while application writes accumulate in a fresh, independent dirty set. There's no interleaving — the two sets never interact. The next flush cycle picks up anything that was written during the previous flush. The worst case is that a value is flushed one cycle late (e.g., "v1" flushes at t=3, "v2" flushes at t=6), but nothing is ever lost.

❌ SMEMBERS + SREM
  • Dirty set is live during flush — reads and writes interleave
  • SREM can remove keys re-dirtied mid-flush → data loss
  • No isolation between flush worker and application threads
  • Bug is timing-dependent → hard to reproduce, appears in production under load
✅ RENAME (Atomic Swap)
  • Single atomic operation creates a frozen snapshot
  • New writes go to a fresh set — complete isolation
  • Worst case: value flushed one cycle late (not lost)
  • No timing dependency — correct under any interleaving

Eviction Policies

When cache is full, eviction policies decide what to remove. Small hit rate differences → enormous DB load differences.

LRU — Least Recently Used

LRU

Evicts entry not accessed for longest time. Hash map (O(1) lookup) + doubly-linked list (O(1) move-to-head, O(1) remove-tail).

Java — LRU Cache (Interview Implementation)
class LRUCache {
    private final int cap;
    private final Map<Integer, Node> map = new HashMap<>();
    private final Node head = new Node(0, 0);  // MRU sentinel
    private final Node tail = new Node(0, 0);  // LRU sentinel

    LRUCache(int cap) {
        this.cap = cap;
        head.next = tail; tail.prev = head;
    }

    int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node n = map.get(key);
        remove(n);
        addFront(n);
        return n.val;
    }

    void put(int key, int val) {
        if (map.containsKey(key)) remove(map.get(key));
        Node n = new Node(key, val);
        map.put(key, n);
        addFront(n);
        if (map.size() > cap) {
            Node lru = tail.prev;
            remove(lru);
            map.remove(lru.key);
        }
    }

    private void remove(Node n) { n.prev.next=n.next; n.next.prev=n.prev; }
    private void addFront(Node n) {
        n.next=head.next; n.prev=head;
        head.next.prev=n; head.next=n;
    }

    static class Node {
        int key, val; Node prev, next;
        Node(int k, int v) { key=k; val=v; }
    }
}
⚠️ LRU's Weakness: Scan Pollution

Sequential scan fills LRU with one-time entries, evicting hot data. Redis uses approximated LRU (sampling 5 random keys) — more resistant and zero memory overhead.

LFU — Least Frequently Used

LFU

Evicts entry with lowest access count. Resists scan pollution. Has frequency drift: old popular entries become un-evictable. Fix: aging or TinyLFU count-min sketch.

Java — O(1) LFU with Frequency Buckets
class LFUCache {
    int cap, minFreq;
    Map<Integer,Integer> keyToVal = new HashMap<>();
    Map<Integer,Integer> keyToFreq = new HashMap<>();
    Map<Integer,LinkedHashSet<Integer>> freqToKeys = new HashMap<>();

    LFUCache(int cap) { this.cap=cap; }

    private void touch(int key) {
        int f = keyToFreq.get(key);
        freqToKeys.get(f).remove(key);
        if (freqToKeys.get(f).isEmpty() && f==minFreq) minFreq++;
        keyToFreq.put(key, f+1);
        freqToKeys.computeIfAbsent(f+1, k->new LinkedHashSet<>()).add(key);
    }

    int get(int key) {
        if (!keyToVal.containsKey(key)) return -1;
        touch(key);
        return keyToVal.get(key);
    }

    void put(int key, int val) {
        if (cap <= 0) return;
        if (keyToVal.containsKey(key)) {
            keyToVal.put(key,val); touch(key); return;
        }
        if (keyToVal.size() >= cap) {
            // Evict LFU (tie-break: LRU within same freq via LinkedHashSet)
            Iterator<Integer> it = freqToKeys.get(minFreq).iterator();
            int evict = it.next(); it.remove();
            keyToVal.remove(evict); keyToFreq.remove(evict);
        }
        keyToVal.put(key,val); keyToFreq.put(key,1);
        freqToKeys.computeIfAbsent(1, k->new LinkedHashSet<>()).add(key);
        minFreq = 1;
    }
}

Event-Driven Invalidation (CDC)

CDC-Based Invalidation

Change Data Capture reads the DB WAL and publishes change events. Catches all writes including direct SQL and other services. Latency: 100ms–2s.

How CDC Works — From WAL to Cache Invalidation

Change Data Capture (CDC)

A technique that captures row-level changes (INSERT, UPDATE, DELETE) from a database's internal transaction log and publishes them as a stream of events. Instead of modifying application code to emit events on every write, CDC taps into the database's own commit mechanism — guaranteeing it captures every change regardless of which service, script, or admin console made the write. The database doesn't know or care that CDC is watching; it's a passive reader of the log the DB already maintains for its own crash recovery.

Write-Ahead Log (WAL)

Before PostgreSQL modifies an actual data page on disk, it first writes a description of the change to a sequential, append-only log file — the Write-Ahead Log. This guarantees that even if the server crashes mid-write, the DB can replay the WAL on startup to recover to a consistent state. The WAL contains every committed change in order: which table, which row, old values, new values, transaction ID, and timestamp. CDC tools like Debezium read this same log — they're essentially "replaying" the DB's own recovery stream as a real-time event feed.

The CDC Pipeline

Why CDC Instead of Application-Level Invalidation?

The naive approach is to add cache invalidation code to every service that writes to the database: after each UPDATE, issue a jedis.del(). This works for a single service but breaks down in real production systems. CDC solves the fundamental problem: you cannot control all write paths to your database.

✅ CDC (WAL-Based)
  • Catches every write — direct SQL, migrations, batch jobs, other microservices, admin consoles. Zero writes can bypass it because it reads the DB's own commit log.
  • Zero code changes in writers — services write to DB normally. No cache library dependency, no invalidation code, no coordination.
  • Guaranteed ordering — WAL events are in commit order. No race between two services invalidating the same key.
  • Decoupled — if the invalidation consumer is down, events queue in Kafka. On recovery, it catches up. No data loss.
  • Derived key invalidation — the consumer can map one DB change to multiple cache keys (e.g., order insert → invalidate user order count, product stock, dashboard cache).
❌ Application-Level Invalidation
  • Missed write paths — a DBA runs UPDATE directly, a batch job modifies rows, another service uses its own connection → cache stays stale forever.
  • Code duplication — every writer service must implement invalidation. One missed path = permanent inconsistency.
  • Race conditions — two services write the same row; invalidation order depends on which DEL arrives first, not which COMMIT was last.
  • Tight coupling — every writer imports the cache client library. Schema changes require coordinated deploys.
  • Fire-and-forget risk — if the DEL call fails (Redis timeout), the stale entry persists until TTL. No retry mechanism by default.
🎯 Interview Framing

When an interviewer asks "how do you keep the cache consistent?", start with TTL as the baseline, then elevate to CDC: "TTL gives us bounded staleness, but for tighter consistency we use CDC via Debezium — it reads the PostgreSQL WAL, publishes change events to Kafka, and our invalidation consumer issues DEL commands to Redis. This catches all writes including those from other services, batch jobs, and direct SQL — no code changes needed in any writer." This answer covers the mechanism, why it's better than app-level, and the specific tools — all in two sentences.

Debezium

An open-source CDC platform built on Kafka Connect. It creates a logical replication slot in PostgreSQL (or equivalent in MySQL/MongoDB), reads the WAL stream, converts each change into a structured JSON event, and publishes it to a Kafka topic (one topic per table by default). Debezium handles snapshotting (initial full table read), schema evolution, and exactly-once delivery semantics. It's the de facto standard for CDC in the Kafka ecosystem.

Logical Replication Slot

A PostgreSQL mechanism that reserves a position in the WAL stream for a consumer. The DB guarantees it won't recycle WAL segments that the slot hasn't consumed yet — so if Debezium goes offline for an hour, no events are lost. The slot resumes from where it left off. Caveat: if the consumer stays offline too long, unreplayed WAL accumulates on disk and can fill the volume — production CDC pipelines need monitoring for replication lag.

Your Consumer = the Action — it plays back the tape and decides what to do: delete this cache key, invalidate that aggregate, broadcast to L1. The camera and tape are infrastructure; the action is your business logic.

Versioned Cache Keys

Instead of invalidating a cache key (deleting it), you change the key itself by incrementing a version counter stored separately in Redis. Readers always build the key using the current version number, so they never see old versions. Old keys become "orphaned" — no one reads them, and they're eventually cleaned up by LRU eviction or their TTL expiring. The key insight: you never have to coordinate deletion with population because old and new versions live under different keys entirely.

The Problem Versioned Keys Solve

Simple cache invalidation (DEL) has a subtle race condition that can leave the cache permanently stale. It happens when an invalidation and a cache population interleave:

The Classic Invalidation Race

Thread A reads a stale value from DB (before Thread B's update commits). Thread B updates DB and DELetes the cache key. Thread A — still running with the old value — writes it back to the cache. Now the cache has the old value, the DB has the new value, and no one will invalidate the cache again because the write already happened. The stale entry persists until TTL expires.

Compare this to simple DEL: both threads operate on the same key user:42, so the stale write at t=4 overwrites the invalidation. With versioned keys, each version gets its own key — writes can never collide across versions.

Orphaned Key Cleanup

Orphaned Keys

When a version is bumped (e.g., v1 → v2), the old key user:42:v1 becomes "orphaned" — it still exists in Redis but no reader will ever request it. These keys consume memory until they're cleaned up. Three cleanup mechanisms work together: (1) TTL expiry — the data TTL (e.g., 3600s) eventually removes the key automatically. (2) LRU eviction — since orphaned keys are never accessed, they're the first candidates for eviction when Redis needs memory. (3) Active cleanup — optionally, the invalidation path can issue a background DEL user:42:v1 after bumping to v2, but this is an optimization, not a correctness requirement.

Trade-offs: Versioned Keys vs Simple DEL

✅ Versioned Keys
  • Race-condition proof — stale writes go to old version keys that nobody reads
  • No coordination needed — reader and writer never touch the same key
  • Works with any cache-aside pattern — no locks, no singleflight, no Lua scripts
  • INCR is O(1) and atomic — single Redis command, no multi-step invalidation
❌ Costs
  • Extra Redis call on read — every read must first GET the version key, then GET the data key (2 round-trips instead of 1). Mitigate with MGET or Redis pipeline.
  • Orphaned key memory — old versions consume memory until TTL/LRU cleanup. For write-heavy keys, this can be significant.
  • Version counter persistence — the version key itself must never expire (or have a very long TTL), otherwise readers fall back to version "1" and might see stale data from an old v1 key.
  • Complexity — more moving parts than simple DEL. Only justified when the race condition actually matters (high-concurrency read+write on the same keys).
🎯 When to Use Versioned Keys vs Simple DEL

Use simple DEL when: writes are infrequent, TTL is short (staleness self-heals quickly), or the data isn't critical (e.g., recommendation scores, analytics dashboards).

Use versioned keys when: high-concurrency reads AND writes on the same key, the data must be consistent after write (e.g., user profile, inventory count), and TTL-based self-healing isn't fast enough for your SLA.

In an interview, mention versioned keys as an escalation from simple DEL: "We start with cache-aside + DEL on write. If the interviewer pushes on the invalidation race condition, we upgrade to versioned keys — each version gets its own key, so stale writes to old versions are harmless."

Classic race: Thread A reads stale value at t=1, Thread B invalidates at t=2, Thread A writes stale to cache at t=3 → permanent stale. With versioned keys, A writes to v1, but B bumped to v2. Next reader fetches v2 → miss → loads fresh. A's stale v1 write is harmless.

Distributed Caching

A single Redis server can hold ~25GB and handle ~100K ops/sec. When your data exceeds one machine's memory or your traffic exceeds one machine's CPU, you need multiple cache nodes working together. This introduces three hard problems that don't exist with a single node.

"Still not able to understand what these patterns are and how they operate in distributed caching."

The Three Problems of Multi-Node Caching

Imagine you have 10 million cache keys and one Redis server can hold 5 million. You need at least 2 servers. Now every GET/SET must answer a new question that didn't exist before: which server holds this key?

The Routing Problem

With N cache nodes, every read and write must deterministically route to the same node for a given key. If GET user:42 goes to Node A but the previous SET user:42 went to Node B, you'll always get a cache miss. The routing must also handle nodes being added (scale up) or removed (crash) without losing all cached data. This is what Consistent Hashing solves.

The Stampede Problem

With one server and one application instance, a cache miss means one DB query. With 20 application instances sharing a distributed cache, a hot key expiring means 20 simultaneous DB queries — all 20 instances see the miss at the same instant and all race to reload from DB. With 10,000 requests/sec on that key, the spike is catastrophic. This is what Cache Stampede prevention solves.

The Hot Key Problem

Consistent hashing ensures each key goes to one specific node. But what if one key gets 100× more traffic than others (a viral post, a flash sale)? All that traffic hits one node while the other nodes sit idle. Adding more nodes doesn't help — the hash still routes that key to the same single node. This is what Hot Key mitigation solves.

🎯 How They Connect

Think of it as a pipeline: Consistent Hashing answers "which node?" → Stampede Prevention answers "how many DB queries on miss?" → Hot Key Mitigation answers "what if one node gets all the traffic?" Each solves a different failure mode, and production systems need all three.

Consistent Hashing

Consistent Hashing

Hash ring (0 to 2^32−1). Keys stored on first node clockwise. Add/remove node → only K/N keys rehash. Virtual nodes (100–200 per physical) ensure even distribution.

The Problem

You have 10 million cached keys and 3 Redis servers. Every time your app does GET user:42, it needs to know which of the 3 servers holds that key. The rule must be:

  • Deterministic — same key always goes to the same server
  • Balanced — keys spread evenly across servers
  • Stable — adding/removing a server doesn't scramble everything

The Naive Way: Modulo

The simplest rule: server = hash(key) % N

Modulo Hashing — Works Until You Scale
// With 3 servers:
hash("user:42") = 73291
73291 % 3 = 2  →  Server C

hash("user:99") = 54820
54820 % 3 = 1  →  Server B

// Add a 4th server — everything breaks:
73291 % 4 = 1  →  Server B   ← was on C! MISS!
54820 % 4 = 0  →  Server A   ← was on B! MISS!
⚠️ Why Modulo Fails at Scale

Changing from 3→4 servers remaps ~75% of all keys. With 10M keys, that's 7.5M sudden cache misses hitting the DB at once — instant outage.

Consistent Hashing: The Ring

Instead of % N, imagine a circular number line (a ring) from 0 to 999.

Step 1 — Place servers on the ring by hashing their names:

Step 1: Hash Servers onto the Ring
hash("Server-A") = 100  →  position 100
hash("Server-B") = 400  →  position 400
hash("Server-C") = 750  →  position 750

Step 2 — Place keys on the ring by hashing them:

Step 2: Hash Keys onto the Ring
hash("user:42") = 250  →  position 250

Step 3 — Find the server: walk clockwise from the key's position until you hit a server.

Step 3: Walk Clockwise to Find Owner
user:42 is at 250.
Walking clockwise: 251, 252, ... 400 → Server B!
∴ user:42 lives on Server B.

That's the entire algorithm. One rule: walk clockwise, stop at the first server.

Why It Survives Server Changes

Adding Server D at position 550

Before D joins, keys at positions 401–750 all went to Server C (next clockwise). After D joins at 550, keys at 401–550 now hit D first (before reaching C). Keys at 551–750 still go to C. Keys at 0–400 and 751–999 are completely untouched.

Only keys in the 401–550 range moved. That's roughly 1/4 of the ring — not 75%.

Server B crashes (removed from ring)

Before crash, keys at 101–400 went to B. After crash, those keys walk clockwise past B's old position and land on the next server (C at 750). Keys assigned to A and C don't change at all.

Only B's keys moved — roughly 1/3 of the ring. The rest is stable.

The Imbalance Problem (Why Virtual Nodes)

With only 3 points on the ring, the arcs are uneven:

Uneven Distribution — Lucky vs Unlucky Hash Values
// Decent distribution:
A at 100: owns range 751–100 = 350 positions (35%)
B at 400: owns range 101–400 = 300 positions (30%)
C at 750: owns range 401–750 = 350 positions (35%)

// Unlucky hash values:
A at 50:  owns 851–50  = 200 positions (20%)
B at 150: owns 51–150  = 100 positions (10%)  ← starving!
C at 850: owns 151–850 = 700 positions (70%)  ← overloaded!

C holds 7× more keys than B. Virtual nodes fix this: instead of 1 point per server, place 150 points per server. Hash "Server-A-vnode-0", "Server-A-vnode-1", ..., "Server-A-vnode-149". Now there are 450 evenly scattered points on the ring. The law of large numbers ensures each server owns ~33% ± 2%.

The virtual nodes for each server are scattered all over the ring, interleaved with each other — not clustered together:

Virtual Nodes — Interleaved Across the Ring
Position 5   → Server A (vnode-23)
Position 10  → Server B (vnode-71)
Position 48  → Server C (vnode-12)
Position 102 → Server A (vnode-0)
Position 155 → Server B (vnode-44)
Position 210 → Server C (vnode-99)
...
Position 550 → Server B (vnode-8)
Position 750 → Server A (vnode-67)
...

Each server owns many small arcs scattered around the ring instead of one big arc. The total of all those small arcs ≈ 1/3 of the ring.

Virtual nodes also fix the crash cascade: when B dies, its 150 scattered points are between A and C points everywhere around the ring, so the freed keys distribute roughly evenly to both A and C — instead of hammering one neighbor.

🎯 Analogy

1 point per server = 3 big slices of pizza (uneven cuts likely). 150 points per server = 450 tiny interleaved slices — even if a few are bigger, each server's total share averages out.

In Code (Java)

Java — Consistent Hashing with Virtual Nodes (TreeMap)
TreeMap<Long, String> ring = new TreeMap<>();

// Add server with 150 virtual nodes
void addNode(String server) {
    for (int i = 0; i < 150; i++) {
        long pos = hash(server + "-vnode-" + i);
        ring.put(pos, server);
    }
}

// Find server for a key: walk clockwise
String getNode(String key) {
    long pos = hash(key);
    Map.Entry<Long, String> entry = ring.ceilingEntry(pos);  // first >= pos
    if (entry == null) entry = ring.firstEntry();  // wrap around ring
    return entry.getValue();
}

TreeMap.ceilingEntry(pos) finds the first position ≥ the key's hash — that's the "walk clockwise" operation in O(log N). If we pass the last node, firstEntry() wraps back to position 0.

Cache Failures

Three distinct failure modes interviewers ask by name. Knowing the taxonomy is a major differentiator.

Cache Penetration

Cache Penetration

Requests for keys that will never exist in DB (attacker IDs). Every request bypasses cache AND DB returns nothing. Cache provides zero protection.

Bloom Filter Deep Dive

Bloom Filter

A space-efficient probabilistic data structure that answers: "Is this element definitely NOT in the set?" It uses a bit array of m bits and k independent hash functions. To add an element, hash it with all k functions and set those bit positions to 1. To check membership, hash the query and verify all k positions are 1. If any position is 0 → definitely not in set (zero false negatives). If all are 1 → probably in set (possible false positive, because other elements may have set those bits).

Bloom Filter Sizing — Worked Example
Total valid keys (n): 10,000,000
Desired false positive rate (p): 1%

Optimal bits per element: m/n = −(ln p) / (ln 2)² = −(ln 0.01) / 0.4805 ≈ 9.58 bits
Total bits: 10M × 9.58 = 95.8M bits = ~12 MB
Optimal hash functions: k = (m/n) × ln 2 = 9.58 × 0.693 ≈ 7 hash functions

Result: 12 MB of memory protects against unlimited penetration attacks
At 0.1% FPR: ~14.4 bits/element = ~18 MB (still tiny)
Java — Bloom Filter for Cache Penetration Defense
// Initialize on startup: load all valid PKs into filter
BloomFilter<String> validKeys = BloomFilter.create(
    Funnels.stringFunnel(UTF_8),
    10_000_000,   // expected insertions
    0.01          // 1% false positive rate → ~12 MB
);

// On startup: bulk-load from DB
jdbcTemplate.query("SELECT id FROM users", rs -> {
    validKeys.put("user:profile:" + rs.getString("id"));
});

// On each cache read: check Bloom BEFORE touching cache or DB
public User getUser(String key) {
    if (!validKeys.mightContain(key)) {
        return null;  // definitely not in DB → skip cache + DB entirely
    }
    // Normal cache-aside path...
    return cacheAsideGet(key);
}

// On new inserts: add to filter via CDC listener
public void onNewUser(String userId) {
    validKeys.put("user:profile:" + userId);
}
⚠️ Bloom Filter Limitations

No deletions — standard Bloom filters don't support removing elements. If a user is deleted from DB, the filter still says "might exist" (a benign false positive — the cache-aside path returns null from DB and sets a negative cache entry). If deletions are frequent, use a Counting Bloom Filter (4 bits per position instead of 1, supports decrement). Rebuild periodically — run a nightly job to rebuild from the DB to clear accumulated false positive drift.

Cache Breakdown

Cache Breakdown

A single hot key expires, causing a burst of concurrent cache misses that all hit the DB simultaneously. Unlike penetration (data doesn't exist), the data does exist — the problem is a momentary gap between expiry and re-population. This is the same phenomenon as cache stampede, but "breakdown" specifically emphasizes the single-key focus.

When discussing breakdown, present solutions as an escalation: "For most keys, TTL with jitter is fine. For hot keys, we add refresh-ahead so they never expire. For ultra-hot keys (flash sales), we remove TTL entirely and rely on CDC for invalidation. The lock is our safety net across all tiers." This shows you can calibrate solutions to the severity of the problem.

Cache Avalanche

Cache Avalanche

A mass failure where large portions of the cache become unavailable simultaneously — either because many keys expire at the same time (synchronized TTL) or because a cache node crashes. The DB is suddenly hit with 100× the normal load, overwhelming its connection pool, causing timeouts that cascade into a full system outage.

Circuit Breaker Pattern

A state machine with three states: Closed (normal, all requests pass through), Open (DB is unhealthy, all requests return fallback/error immediately without touching DB), and Half-Open (allow one probe request through; if it succeeds, close the circuit; if it fails, re-open). Resilience4j and Hystrix implement this. The circuit breaker prevents your cache miss path from pile-driving a DB that's already failing.

Java — Circuit Breaker on Cache Miss Path (Resilience4j)
CircuitBreaker cb = CircuitBreaker.ofDefaults("cache-miss-db");

public User getUserWithProtection(String key) {
    // Check cache first
    String cached = jedis.get(key);
    if (cached != null) return deserialize(cached);

    // Cache miss → DB query is circuit-breaker protected
    try {
        return cb.executeSupplier(() -> {
            User u = db.query(key);
            jedis.setex(key, ttlWithJitter(), serialize(u));
            return u;
        });
    } catch (CallNotPermittedException e) {
        // Circuit OPEN → DB is overwhelmed → return degraded response
        log.warn("Circuit open, returning fallback for {}", key);
        return User.FALLBACK;  // cached stale version, default, or 503
    }
}

private long ttlWithJitter() {
    return 3600 + ThreadLocalRandom.current().nextInt(360);  // 10% jitter
}
⚠️ Avalanche vs Breakdown — The Scale Difference

Breakdown is a single hot key expiring — it causes a localized DB spike that might be survivable. Avalanche is thousands of keys expiring simultaneously or an entire cache node dying — it causes a system-wide DB overload that cascades into full outage within seconds. The defenses are different: breakdown needs a lock or refresh-ahead on individual keys; avalanche needs infrastructure-level protection (jitter, HA, circuit breaker, rate limiting).

Penetration: non-existent keys → Bloom filter. Breakdown: hot key expiry → lock/refresh-ahead. Avalanche: mass failure → jitter + circuit breaker. Name all three in the interview.

L1 — In-Process Cache

Caffeine (JVM) · groupcache (Go)

The fastest possible cache: a hash map in the application's own heap memory. Zero serialization, zero network, zero system calls. The latency difference between L1 and L2 is 3–4 orders of magnitude: ~50ns vs ~2ms.

Why L1 Matters — The Latency Gap

A Redis GET takes ~1–3ms (TCP round-trip within the same AZ). A Caffeine lookup takes ~50 nanoseconds (hash map access in local heap). That's a 40,000× difference. For a service handling 50K requests/sec, if L1 absorbs 60% of reads, you eliminate 30K Redis round-trips per second — saving 30K × 2ms = 60 seconds of cumulative latency per second. L1 is the single biggest latency win in the entire caching stack.

Each JVM instance has its own L1 cache — there's no shared state between instances. When Service A updates a key, Service A's L1 is updated, but Service B/C/D still serve their stale L1 copies. Three strategies to handle this:

// Publisher side: after L2 invalidation, broadcast to all L1s
public void invalidate(String key) {
    jedis.del(key);                          // L2 invalidation
    jedis.publish("cache:invalidate", key);  // broadcast to L1s
}

// Subscriber side: each instance listens and evicts locally
jedis.subscribe(new JedisPubSub() {

}, "cache:invalidate");
⚠️ Pub/Sub Is Fire-and-Forget

Redis Pub/Sub has no delivery guarantee. If an instance is briefly disconnected (GC pause, network blip), it misses the invalidation message and serves stale L1 data. This is why L1 always needs a short TTL as a backup — even with pub/sub, the TTL ensures staleness is bounded. The pub/sub makes it faster (5ms vs 60s), the TTL makes it safe (bounded worst case).

L2 — Distributed Cache

Redis Cluster / Memcached

Shared across all application instances. When one instance populates a key, all other instances benefit. The 1–3ms network round-trip is the dominant cost — but it's still 5–10× faster than a DB query (~15ms).

Redis Cluster Topology

Redis Cluster — Hash Slots

Redis Cluster partitions the keyspace into 16,384 hash slots. Each key is mapped to a slot via CRC16(key) % 16384. Each primary node owns a contiguous range of slots (e.g., Node A owns slots 0–5460, Node B owns 5461–10922, Node C owns 10923–16383). When you add a node, Redis migrates a subset of slots (and their keys) to the new node — similar in spirit to consistent hashing but with explicit slot assignment instead of a virtual ring.

The L1 + L2 Interaction — Complete Flow

🎯 Read Path: L1 → L2 → DB

Step 1: Check L1 (Caffeine). Hit → return in ~50ns. Done.

Step 2: L1 miss → Check L2 (Redis GET). Hit → promote to L1 (so next read is 50ns), return in ~2ms.

Step 3: L2 miss → Query DB (~15ms). Write result to L2 (shared — all instances benefit) → write to L1 (local — this instance benefits). Return.

🔑 Write Path: DB → Invalidate L2 → Broadcast L1

Step 1: Write to DB (source of truth first).

Step 2: DEL the L2 key in Redis.

Step 3: PUBLISH invalidation event → all instances evict from their L1.

This three-step sequence ensures: DB is always authoritative, L2 doesn't serve stale data after the DEL, and L1 instances converge within pub/sub latency (~5ms).

CDN Edge Cache

CloudFront / Fastly / Cloudflare

Caches HTTP responses at edge PoPs (Points of Presence) geographically close to users. The CDN is the outermost cache tier — it sits between the user's browser and your API Gateway. For static assets and public API responses, the CDN can serve 90%+ of traffic without any request reaching your origin infrastructure.

HTTP Cache Headers — The Control Plane

CDN caching is controlled entirely via HTTP response headers. Your origin server sets these headers; the CDN obeys them.

Cache-Control Header

The primary directive. max-age=300 tells the browser to cache for 5 minutes. s-maxage=300 tells the CDN specifically (overrides max-age for shared caches). public means any cache (CDN, browser) can store it. private means only the browser (not CDN) should cache — use for user-specific data.

stale-while-revalidate

stale-while-revalidate=60 tells the CDN: "After the content expires, you can serve the stale copy for up to 60 more seconds while you fetch a fresh version from origin in the background." This eliminates CDN cache miss latency for users — they always get an instant response. The content might be up to 60 seconds stale during revalidation, but no user ever waits for the origin round-trip.

Cache Warming (Pre-Heating)

Cache Warming

Proactively populating the cache before it starts serving production traffic. On a cold start (new deployment, new region, cache flush after a Redis upgrade), the cache has zero entries — every request is a miss, creating a temporary avalanche on the DB. Cache warming eliminates this cold-start penalty by pre-loading the keys most likely to be requested.

When Cold Starts Happen

Cold starts are more common than most people realize. They occur on every deployment (new JVM instance has empty L1), after Redis maintenance (cluster restart flushes all data), when scaling out (new instances join with empty caches), and during failover (replica promoted with stale/partial data). A system doing 50 deployments/week experiences 50 cold starts — each one is a potential DB spike if not handled.

Java — Cache Warming on Application Startup
@Component
public class CacheWarmer implements ApplicationRunner {
    private final JdbcTemplate db;
    private final Jedis jedis;
    private final ObjectMapper mapper;
    private final ThreadLocalRandom rng = ThreadLocalRandom.current();

    @Override
    public void run(ApplicationArguments args) {
        log.info("Warming cache with top 10K products...");

        // Strategy: Eager DB Scan for hot data
        List<Product> hot = db.query(
            "SELECT * FROM products ORDER BY view_count DESC LIMIT 10000",
            productRowMapper);

        Pipeline p = jedis.pipelined();
        for (Product prod : hot) {
            String key = "product:detail:" + prod.getId();
            // TTL with jitter — CRITICAL to prevent synchronized expiry
            long ttl = 3600 + rng.nextInt(360);  // 3600 ± 360s
            p.setex(key, ttl, mapper.writeValueAsString(prod));
        }
        p.sync();

        log.info("Cache warm complete: {} keys loaded", hot.size());
    }
}
🔑 Warming + Jitter = Safe Cold Start

Always add random TTL jitter when warming: TTL = base_ttl + random(0, base_ttl × 0.1). Without jitter, all 10,000 warmed keys expire at the same time (exactly base_ttl seconds after deployment), creating a secondary avalanche. With jitter, expirations are spread over a 6-minute window (10% of 3600s), and refresh-ahead keeps the hottest keys alive indefinitely.

🎯 Interview Tip: Warming as Cold Start Defense

When discussing avalanche prevention, mention warming proactively: "On deployment, we warm the cache with the top 10K keys from yesterday's access logs before the instance joins the load balancer. Each key gets a randomized TTL to prevent synchronized expiry. Between warming and refresh-ahead, the effective cold start window is near zero."

🔑 The Pattern: Match Strategy to Access Pattern

The decision tree is: (1) Is it read-heavy or write-heavy? Read-heavy → cache-aside or read-through. Write-heavy → write-back or write-around. (2) Do you need read-after-write consistency? Yes → write-through. (3) Is it a hot key? → Add refresh-ahead (L1) and key replication (L2). (4) Is it static? → CDN. There's no "best" strategy — there's the right strategy for each data class.
