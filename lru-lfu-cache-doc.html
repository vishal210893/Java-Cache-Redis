Caching is the backbone of every high-performance system — from CPU L1/L2 caches to CDN edge nodes to application-layer Redis clusters. The two most critical eviction policies, LRU (Least Recently Used) and LFU (Least Frequently Used), appear in nearly every systems interview. This document covers their internals, implementation, tradeoffs, pitfalls, and the state-of-the-art hybrid approaches that production systems actually use.

🏢 Where This Comes Up

Meta, Google, Amazon, Apple, Microsoft — virtually every company asks cache design questions. LRU (LeetCode 146) is one of the top 5 most frequently asked coding questions across all FAANG interviews. LFU (LeetCode 460) is a common hard-level follow-up.

Cache Eviction Policies

Cache Eviction Policy

A strategy that determines which entry to remove from a cache when the cache is at capacity and a new entry needs to be inserted. The policy defines the decision function — which existing entry is the "least valuable" and should be sacrificed.

A cache has limited capacity. When full and a new entry must be inserted, the cache must decide which existing entry to evict. The eviction policy defines this decision. The choice of policy fundamentally determines cache hit rates and therefore system performance.

🎯 Interview Focus

LRU and LFU are the two most frequently asked in interviews. LRU tests fundamental data structure fluency (HashMap + DLL), while LFU tests the ability to design a more complex O(1) system with frequency tracking. Know both cold — implementation, tradeoffs, pitfalls, and when to use which.

Requirements

Functional Requirements

get(key) → value

Retrieve the value associated with a key. Returns -1 on cache miss. Must update recency (LRU) or frequency (LFU) metadata on every access.

put(key, value)

Insert or update a key-value pair. If the cache is at capacity and the key is new, evict one entry according to the eviction policy before inserting.

evict()

Remove the least valuable entry. LRU: remove the entry not accessed for the longest time. LFU: remove the entry with the lowest access frequency (tie-break by LRU).

O(1) All Operations

Both get() and put() (including eviction) must run in constant time. This drives the choice of data structures — HashMap for lookup, DLL for ordering.

Thread Safety (Extended)

Support concurrent access from multiple threads. Must handle the subtlety that LRU's get() is a write operation (mutates ordering).

TTL Support (Extended)

Per-entry Time-To-Live. Expired entries treated as cache misses on access. Background sweep for memory reclamation.

Data Model

Each cache implementation composes a small set of data structures to achieve O(1) for all operations. The choice of structures is the core design insight that interviewers are testing.

LRU Data Structures

HashMap
O(1) Key Lookup

• Maps keys → DLL Nodes
• Provides instant access to any cache entry without traversal
• Pre-sized to capacity / 0.75 + 1 to avoid rehashing

Doubly Linked List
O(1) Order Maintenance

• Maintains access recency order
• Head = most recently used (MRU)
• Tail = least recently used (LRU)
• Sentinel head/tail nodes eliminate null checks
• Supports O(1) insert, remove, and move-to-head

Sentinel Node (Dummy Node)

A placeholder node at the head and tail of a doubly linked list that never holds real data. Sentinels eliminate all null-pointer checks in insert/remove operations because every real data node is guaranteed to have non-null prev and next pointers. This simplifies code and eliminates boundary-condition bugs.

Why a Doubly Linked List?

❌ Singly Linked List

To remove a node, you must find its predecessor — which requires O(n) traversal from the head. Even if the HashMap gives you a direct pointer to the node, you can't unlink it without knowing what comes before it.

✅ Doubly Linked List

Each node stores a prev pointer. Removal is just 2 pointer assignments:
node.prev.next = node.next
node.next.prev = node.prev
O(1) regardless of list position. This is why DLL is non-negotiable for LRU.

If asked "can you implement LRU without a DLL?", walk through this table. Show you know the alternatives and why each falls short. Then land on: "HashMap + DLL is the only combination that achieves O(1) for all three operations."

LFU Data Structures

Key Map (HashMap)
O(1) Key → Node Lookup

• Maps each key → its DLL Node
• Identical role to LRU's HashMap
• Provides instant access without scanning frequency buckets

Frequency Map (HashMap)
Frequency → DLL Buckets

• Maps each frequency count → a Doubly Linked List
• Groups all nodes at the same access frequency
• Within each bucket, nodes ordered by recency (MRU at head) for tie-breaking

minFrequency Tracker
O(1) Eviction Target

• A single integer tracking the current minimum frequency
• Enables O(1) eviction — jump directly to lowest-frequency bucket
• Updated on insert (reset to 1) and on promotion from min bucket (increment by 1)
• Never needs to decrease or skip values

Node Structure

✅ LRU Node

key — needed to remove from HashMap on eviction

value — the cached data

prev — pointer to previous node in DLL

next — pointer to next node in DLL

Overhead: ~16 bytes (two pointers on 64-bit)

✅ LFU Node

key — needed to remove from KeyMap on eviction

value — the cached data

frequency — access count (starts at 1)

prev — pointer to previous node in freq bucket DLL

next — pointer to next node in freq bucket DLL

Overhead: ~20 bytes (two pointers + one int)

⚠️ Why Key Must Be Stored in Node

When evicting the tail node from the DLL, you need to also remove it from the HashMap. Without the key stored in the node, you'd need an O(n) scan of the HashMap to find which key maps to this node. Forgetting this is one of the most common implementation bugs.

High-Level Design

LRU Internal Architecture

The LRU cache combines a HashMap with a Doubly Linked List. The HashMap provides O(1) lookup, while the DLL maintains access order with the most recently used entry at the head and the least recently used at the tail.

LFU Internal Architecture

The LFU cache uses a KeyMap for O(1) lookup, a FreqMap that groups nodes by access frequency, and a minFrequency tracker for O(1) eviction. Within each frequency bucket, a DLL maintains recency order for tie-breaking.

LRU — Operation Walkthrough

1
get("B") — Cache Hit
Recency Update (moveToHead)
• HashMap lookup finds Node B in O(1)
• removeNode(B) — update B's neighbors to skip over B
• addToHead(B) — re-insert B immediately after HEAD sentinel
• B is now the MRU entry
• Return B's value
2
put("E", val) — Cache Full
Eviction + Insertion
• Cache is at capacity — must evict before inserting
• Evict: Remove tail.prev from the DLL (the LRU entry)
• Remove evicted key from HashMap (this is why key is stored in Node)
• Insert: Create Node E, add after HEAD sentinel
• Add E to HashMap
• E is now MRU

LFU — Operation Walkthrough

1
get("B") — B has freq=2
Frequency Promotion
• KeyMap lookup finds Node B in O(1)
• Remove B from freqMap[2]'s DLL
• Check minFrequency: Is freqMap[2] now empty AND minFrequency == 2?
   → Yes: increment minFrequency to 3
   → No: minFrequency stays unchanged
• Increment B.frequency from 2 → 3
• Insert B at HEAD of freqMap[3]'s DLL (creates bucket if needed)
• Return B's value
2
put("F", val) — Cache Full
Min-Frequency Eviction + Reset
• Cache is at capacity — must evict before inserting
• Find victim: Go to freqMap[minFrequency]'s DLL
• Evict: Remove tail.prev from that DLL (least frequent; among ties, least recent)
• Remove evicted key from KeyMap
• Insert: Create Node F with frequency = 1
• Add F to freqMap[1]'s DLL at HEAD
• Set minFrequency = 1 — always, because new node has freq=1
• Add F to KeyMap

Deep Dive — LRU Implementation

Complete Java Implementation

Java 
import java.util.HashMap;
import java.util.Map;

public class LRUCache {

    // Doubly linked list node
    static class Node {
        int key;
        int value;
        Node prev;
        Node next;

        Node(int key, int value) {
            this.key = key;
            this.value = value;
        }
    }

    private final int capacity;
    private final Map<Integer, Node> map;

    // Sentinel nodes — eliminates null checks at boundaries
    private final Node head;
    private final Node tail;

    public LRUCache(int capacity) {
        this.capacity = capacity;
        this.map = new HashMap<>();

        // head and tail are dummy sentinels, never removed
        this.head = new Node(-1, -1);
        this.tail = new Node(-1, -1);
        head.next = tail;
        tail.prev = head;
    }

    public int get(int key) {
        Node node = map.get(key);
        if (node == null) {
            return -1; // cache miss
        }
        // Move to head (mark as most recently used)
        moveToHead(node);
        return node.value;
    }

    public void put(int key, int value) {
        Node node = map.get(key);

        if (node != null) {
            // Key exists — update value and move to head
            node.value = value;
            moveToHead(node);
        } else {
            // New key — check capacity
            if (map.size() >= capacity) {
                // Evict LRU entry (tail.prev is the least recently used)
                Node lru = tail.prev;
                removeNode(lru);
                map.remove(lru.key);
            }
            // Insert new node at head
            Node newNode = new Node(key, value);
            addToHead(newNode);
            map.put(key, newNode);
        }
    }

    // --- Internal linked list operations ---

    private void addToHead(Node node) {
        node.prev = head;              // new node's prev → HEAD sentinel
        node.next = head.next;          // new node's next → current first node (NOT tail!)
        head.next.prev = node;          // old first node's prev → new node
        head.next = node;               // HEAD sentinel's next → new node
    }

    private void removeNode(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }

    private void moveToHead(Node node) {
        removeNode(node);
        addToHead(node);
    }
}

Why Sentinel Nodes Matter

⚠️ Common Confusion: head.next ≠ tail

In addToHead(), head.next refers to the current first real data node — NOT the TAIL sentinel. The TAIL sentinel is the last node in the list. head.next == tail is only true when the list is empty. When there are N real nodes between the sentinels, head.next is Node 1 (MRU) and tail.prev is Node N (LRU). Replacing head.next with tail would break insertion by skipping all existing nodes.

❌ Without Sentinels — Error-Prone

Every removeNode() and addToHead() needs null checks for both prev and next. Four branches per operation. Easy to miss an edge case, especially when the list has 0 or 1 real nodes.

✅ With Sentinels — Clean & Bug-Free

Real data nodes always have non-null prev and next (pointing to sentinels at minimum). Zero branches in removeNode() — just 2 pointer assignments. This is why experienced engineers always use sentinels.

Java — Without Sentinels (DON'T DO THIS) 
// WITHOUT sentinels — 4 branches, error-prone
private void removeNode(Node node) {
    if (node.prev != null) {
        node.prev.next = node.next;
    } else {
        head = node.next; // node was head
    }
    if (node.next != null) {
        node.next.prev = node.prev;
    } else {
        tail = node.prev; // node was tail
    }
}

Java's LinkedHashMap Shortcut

Java — LinkedHashMap LRU 
import java.util.LinkedHashMap;
import java.util.Map;

public class LRUCacheLinkedHashMap extends LinkedHashMap<Integer, Integer> {
    private final int capacity;

    public LRUCacheLinkedHashMap(int capacity) {
        // accessOrder=true makes it ordered by access time, not insertion
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<Integer, Integer> eldest) {
        return size() > capacity;
    }
}
🔍 When to Use Which

Interview: Always implement from scratch with HashMap + DLL. This is what interviewers are testing — understanding of the underlying mechanics.

Production: LinkedHashMap is fine for simple cases. For production-grade caching, use Caffeine (Java) or Guava Cache, which handle concurrency, TTL, statistics, and eviction callbacks.

LRU — Strengths & Weaknesses

✅ Strengths
  • Simple mental model — "most recently used stays" is intuitive
  • O(1) everything — get, put, evict all constant time
  • Adapts to workload shifts — when access patterns change, cache adjusts within one full cycle
  • Low overhead — just 2 extra pointers per entry (~16 bytes)
  • Temporal locality — exploits the principle that recently accessed data is likely accessed again
❌ Weaknesses
  • Scan pollution — a sequential scan of N items (N > cache size) flushes the entire cache, evicting hot entries
  • No frequency signal — an item accessed 1,000× and one accessed 1× are treated the same. Most recent wins regardless of history
  • Pathological workload — if working set is slightly larger than cache, LRU can achieve 0% hit rate, cycling through entries just before reuse
Adapts to Workload Shifts — Explained

Imagine your cache (capacity=5) holds hot items A–E. A marketing campaign launches and users suddenly want X, Y, Z, W, V instead.

With LRU, the moment X is accessed it goes to the head. Then Y, then Z. Within just 5 new accesses (one full cache cycle), A–E are entirely replaced by X–V. LRU naturally "forgets" the old pattern because recency is the only signal — no historical baggage.

With LFU, A–E might have accumulated frequencies of 500+. New entries X–Z arrive at freq=1 and are immediately evicted. It could take thousands of accesses before the new set overtakes the old. "One full cycle" = enough unique new accesses to fill the cache once.

Scan Pollution — Explained

A "scan" is when something reads through a large dataset sequentially, touching each item exactly once. Real examples: a database SELECT * FROM users, a nightly analytics job reading every log file, a backup process reading all objects.

Say your cache (capacity=5) has hot items A, B, C. A batch job scans D through K — each accessed once:

Before scan: [C, B, A, _, _] ← hot items safe Scan D: [D, C, B, A, _] Scan E: [E, D, C, B, A] ← cache full Scan F: [F, E, D, C, B] ← A evicted! Scan G: [G, F, E, D, C] ← B evicted! Scan H: [H, G, F, E, D] ← C evicted! After scan: [K, J, I, H, G] ← ALL hot items gone

The scan items will never be accessed again, but they flushed out entries that would have been accessed again. One bad scan destroys your entire useful cache. This is called "pollution" because useless one-time entries contaminate the cache.

Pathological Workload — Explained

A surprisingly common worst-case. Cache capacity = 4, working set = 5 items (A–E), access pattern cycles: A → B → C → D → E → A → B → ...

Access A: [A] MISS Access B: [B, A] MISS Access C: [C, B, A] MISS Access D: [D, C, B, A] MISS (cache full) Access E: [E, D, C, B] MISS (A evicted) Access A: [A, E, D, C] MISS (A was JUST evicted one step ago!) Access B: [B, A, E, D] MISS (same pattern repeats)

Every single access is a miss. 0% hit rate. The working set (5) is barely larger than the cache (4), so every time you return to an item, it was evicted exactly one step ago. You're always one step too late.

The cliff is dramatic: increase cache to 5 → 100% hit rate. Decrease by 1 → 0% hit rate. This discontinuity is the pathological behavior.

Deep Dive — LFU Implementation

Complete Java Implementation

Java 
import java.util.HashMap;
import java.util.Map;

public class LFUCache {

    static class Node {
        int key, value, frequency;
        Node prev, next;
        Node(int k, int v) { key = k; value = v; frequency = 1; }
    }

    static class DoublyLinkedList {
        Node head, tail;
        int size;

        DoublyLinkedList() {
            head = new Node(-1, -1);
            tail = new Node(-1, -1);
            head.next = tail; tail.prev = head;
            size = 0;
        }

        void addToHead(Node n) {
            n.prev = head; n.next = head.next;
            head.next.prev = n; head.next = n;
            size++;
        }

        void removeNode(Node n) {
            n.prev.next = n.next; n.next.prev = n.prev;
            size--;
        }

        Node removeTail() {
            if (size == 0) return null;
            Node lfu = tail.prev;
            removeNode(lfu);
            return lfu;
        }

        boolean isEmpty() { return size == 0; }
    }

    private final int capacity;
    private int minFrequency;
    private final Map<Integer, Node> keyMap;
    private final Map<Integer, DoublyLinkedList> freqMap;

    public LFUCache(int capacity) {
        this.capacity = capacity;
        this.minFrequency = 0;
        this.keyMap = new HashMap<>();
        this.freqMap = new HashMap<>();
    }

    public int get(int key) {
        Node node = keyMap.get(key);
        if (node == null) return -1;
        updateFrequency(node);
        return node.value;
    }

    public void put(int key, int value) {
        if (capacity == 0) return;

        Node node = keyMap.get(key);
        if (node != null) {
            node.value = value;
            updateFrequency(node);
        } else {
            if (keyMap.size() >= capacity) {
                // Evict least frequent (and among those, least recent)
                DoublyLinkedList minList = freqMap.get(minFrequency);
                Node evicted = minList.removeTail();
                keyMap.remove(evicted.key);
            }
            Node newNode = new Node(key, value);
            keyMap.put(key, newNode);
            freqMap.computeIfAbsent(1, k -> new DoublyLinkedList()).addToHead(newNode);
            minFrequency = 1; // new node always has freq=1
        }
    }

    private void updateFrequency(Node node) {
        int oldFreq = node.frequency;
        DoublyLinkedList oldList = freqMap.get(oldFreq);
        oldList.removeNode(node);

        // If we emptied the min frequency bucket, increment minFrequency
        if (oldFreq == minFrequency && oldList.isEmpty()) {
            minFrequency++;
        }

        // Move to next frequency bucket
        node.frequency++;
        freqMap.computeIfAbsent(node.frequency, k -> new DoublyLinkedList())
               .addToHead(node);
    }
}

Why minFrequency Is Always Correct

🧠 The Key Insight

The minFrequency variable seems fragile — how do we know it's correct without scanning all frequencies? The proof relies on two observations:

minFrequency Correctness Proof

Case 1 — New entry inserted: Always has frequency=1, so minFrequency = 1. Guaranteed correct because no entry can have frequency < 1.

Case 2 — Entry promoted from minFrequency bucket: If the freqMap[minFrequency] bucket becomes empty after promotion, minFrequency++. This works because the promoted node went to minFrequency + 1, which is now the new minimum.

Why it never needs to decrease (except on new insert): Any node at frequency F was previously at frequency F-1. So if F-1's bucket is empty and F is the smallest non-empty bucket, minFrequency = F is correct.

Why it never skips values: Frequencies increment by exactly 1 on each access. There are no gaps. If minFrequency was 3 and we empty bucket 3, the promoted node is in bucket 4 — so minFrequency = 4 is the true minimum.

When get("B") is called and B has freq=2, it must move from the freq=2 bucket to freq=3. If B was the last entry in the minFrequency bucket, minFrequency increments.

LFU — Strengths & Weaknesses

✅ Strengths
  • Protects hot entries — an item accessed 1,000× won't be evicted by a one-time scan
  • Resists scan pollution — scanner entries have low frequency, evicted immediately
  • Stable workloads — when certain items are consistently popular, LFU keeps them cached
❌ Weaknesses
  • Cold start problem — new entries start at freq=1, immediately vulnerable to eviction even if they'll become hot
  • Stale popularity (cache pollution) — historically popular items retain high frequency even when no longer accessed. They resist eviction indefinitely
  • Slow adaptation — if the popular set changes, LFU is slow to reflect it because old items have accumulated frequency
  • Higher memory/complexity — two HashMaps, frequency buckets, minFrequency tracking
Protects Hot Entries — Explained

In LFU, every access increments a frequency counter. An item accessed 1,000 times has freq=1000. A scan item that enters once has freq=1.

When the cache is full, LFU evicts the entry with the lowest frequency. The scan item (freq=1) gets evicted long before the hot item (freq=1000). The hot item's accumulated frequency acts like armor — it would take 999 new entries, each accessed more often, to displace it.

This is the fundamental advantage over LRU, where a single recent access can save any item regardless of history. In LRU, accessing a cold item once makes it "more valuable" than a hot item last accessed 2 seconds ago.

Resists Scan Pollution — Explained

Same scan scenario as LRU's weakness — a batch job reads D, E, F, G, H sequentially. But with LFU:

Before scan: A(freq=100) B(freq=80) C(freq=60) — hot items cached Scan D: D enters with freq=1. Cache full → evict lowest freq. D(freq=1) replaces whatever had the previous min freq. Scan E: E enters with freq=1. D(freq=1) is now the lowest → D evicted. Scan F: F enters with freq=1. E(freq=1) evicted. Scan G: G enters with freq=1. F(freq=1) evicted. After scan: A(freq=100) B(freq=80) C(freq=60) still safe!

The scan items keep evicting each other because they all have freq=1 — the lowest possible. They never accumulate enough frequency to threaten the hot items. The scan rolls through the cache like water off a duck's back.

Slow Adaptation — Explained

This is the flip side of "protects hot entries." The same frequency armor that protects hot items becomes a problem when those items stop being hot.

Imagine product A was trending last week — accessed 10,000 times, so A.freq=10000. This week, product A is dead and product X is the new trend. X starts at freq=1:

After 1 hour: A.freq=10,000 (stale) X.freq=100 → A still "wins" After 10 hours: A.freq=10,000 (stale) X.freq=1,000 → A still "wins" After 100 hours: A.freq=10,000 (stale) X.freq=10,000 → Finally tied

It took 100 hours for the new trend to match the old one's frequency. During all that time, A occupied a cache slot uselessly, and any newer entries with freq < 10,000 would be evicted before stale A.

LRU would have evicted A within seconds of the last access — because recency is the only signal, there's no historical baggage to burn through. This is exactly what W-TinyLFU's periodic frequency decay solves: by halving all counts regularly, old popularity naturally fades.

A(4) B(3) C(3) still safe!
Each scan item enters at freq=1, immediately evicted by the next

Re-access A: HIT → freq=5
Re-access B: HIT → freq=4
Re-access C: HIT → freq=4

Frequency shield blocked the scan entirely.

🎯 Takeaway

LFU's frequency shield protects hot entries from being displaced by a sequential scan. This is LFU's strongest scenario.

Scenario 2 — Stale Popularity (LRU Wins)
Setup Cache capacity = 2 Access pattern: A×5 (A becomes "stale" — never accessed again) then B×10
✅ LRU — Stale Entry Evicted Naturally

After A×5:
[A] — cached

After B×10:

New entry C arrives:

Stale entries age out automatically via recency.

⚠️ Takeaway

LFU's historical frequency count becomes a liability when access patterns shift. Once-popular items occupy cache indefinitely, starving new entries. This is the "frequency pollution" problem.

When to Use Which

✅ Use LRU When
  • Temporal locality — recently accessed items likely accessed again soon
  • Shifting patterns — the "hot set" changes frequently
  • Simple requirements — well-understood, easy to debug
  • Uniform frequency — all items accessed at similar rates

Examples: Web browser cache, OS page cache, DB buffer pool, K8s controller caches

📊 Use LFU When
  • Stable hot set — small set gets vast majority of requests consistently
  • Scan-heavy workloads — periodic batch scans could pollute LRU
  • CDN/content caching — popular content stays popular for extended periods

Examples: CDN edge caches, DNS resolvers, content recommendation caches

Scan-Heavy Workloads — Why LFU Wins Here

"Scan-heavy" means scans aren't rare exceptions but regular, expected operations. Real examples: a data pipeline running hourly aggregations over every record, a search indexer that periodically crawls all documents, a monitoring system sweeping all metrics for alerting.

In these systems, LRU would be flushed on every scan cycle. Your hit rate would crater every hour when the batch job runs, slowly rebuild, then get destroyed again — a sawtooth pattern of terrible performance:

LRU hit rate over time: ████████▁▁████████▁▁████████▁▁ ↑ ↑ ↑ scan 1 scan 2 scan 3 (cache flushed each time, rebuilds slowly) LFU hit rate over time: ██████████████████████████████ (scan items evict each other at freq=1, hot items untouched)

LFU handles this gracefully because scan entries (freq=1) never displace genuinely popular entries. Your cache maintains a stable, high hit rate regardless of scan frequency. This is why CDN edge caches and DNS resolvers lean toward frequency-based eviction — they deal with periodic crawlers and bots that would pollute a recency-based cache.

Deep Dive — Thread Safety & Concurrency

ReadWriteLock Pitfall

A ReadWriteLock allows multiple concurrent readers OR a single writer. The critical insight: LRU's get() is NOT a read-only operation — it modifies the linked list by moving the accessed node to the head. Therefore, both get() and put() require the write lock. Using a read lock for get() allows concurrent list mutations, causing data structure corruption.

Thread-Safe LRU

Java 
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class ConcurrentLRUCache {
    private final LRUCache cache;
    private final ReentrantReadWriteLock lock = new ReentrantReadWriteLock();

    public ConcurrentLRUCache(int capacity) {
        this.cache = new LRUCache(capacity);
    }

    public int get(int key) {
        // IMPORTANT: get() is NOT read-only in LRU!
        // It modifies the linked list (moves node to head).
        // Must use WRITE lock, not read lock.
        lock.writeLock().lock();
        try {
            return cache.get(key);
        } finally {
            lock.writeLock().unlock();
        }
    }

    public void put(int key, int value) {
        lock.writeLock().lock();
        try {
            cache.put(key, value);
        } finally {
            lock.writeLock().unlock();
        }
    }
}

For most applications, a single global lock is sufficient. Java's Caffeine library uses a buffered-write approach inspired by database WAL — access events are recorded in a lock-free ring buffer and a single drain thread applies them to the LRU list periodically. This makes reads nearly lock-free while maintaining correct ordering.

Deep Dive — TTL Integration

Time-To-Live (TTL)

A per-entry expiration time after which the entry is considered stale and should no longer be served. Pure LRU/LFU don't consider time — an entry from 30 days ago with a recent access is treated the same as one from 5 minutes ago. TTL adds temporal correctness to the cache.

TTL Implementation Approaches

Approach 1 — Lazy Expiry (Check on Access)
Java 
public int get(int key) {
    Node node = map.get(key);
    if (node == null) return -1;
    if (System.currentTimeMillis() > node.expiresAt) {
        // Expired — treat as miss, remove
        removeNode(node);
        map.remove(key);
        return -1;
    }
    moveToHead(node);
    return node.value;
}

Pro: Zero background overhead. Con: Expired entries occupy memory until accessed.

Approach 2 — Background Sweep
Java 
ScheduledExecutorService sweeper = Executors.newSingleThreadScheduledExecutor();
sweeper.scheduleAtFixedRate(() -> {
    for (Map.Entry<Integer, Node> entry : map.entrySet()) {
        if (System.currentTimeMillis() > entry.getValue().expiresAt) {
            removeNode(entry.getValue());
            map.remove(entry.getKey());
        }
    }
}, 60, 60, TimeUnit.SECONDS);

Pro: Proactively frees memory. Con: O(n) sweep cost, needs locking, sweep interval is a tuning knob.

✅ Recommended: Both (Lazy + Background)

Use lazy expiry on get() for correctness (never serve stale data) combined with a background sweep for memory reclamation (don't let expired entries accumulate). This is the standard production approach used by Redis, Caffeine, and Memcached.

Deep Dive — Design Decisions & Tradeoffs

Count-Based vs Size-Based Capacity

📐 Count-Based (maxEntries=1000)
  • Simple — evict one entry per insert over capacity
  • Assumes entries are roughly equal size
  • Fails when sizes vary (1KB metadata vs 10MB image)

Use when: entries are same type, similar size (DB rows, config)

📏 Size-Based (maxBytes=256MB)
  • Tracks byte size per entry, evicts in a loop until room
  • Must reject entries exceeding total cache size
  • More accurate memory control

Use when: entries vary in size (HTTP responses, files, blobs)

Java — Size-Bounded Eviction 
public void put(int key, byte[] value) {
    long newSize = value.length;
    if (newSize > maxBytes) return; // reject — don't flush everything

    // Evict LRU entries in a loop until there's room
    while (currentBytes + newSize > maxBytes && !isEmpty()) {
        Node lru = tail.prev;
        currentBytes -= lru.size;
        removeNode(lru);
        map.remove(lru.key);
    }

    Node node = new Node(key, value, newSize);
    addToHead(node);
    map.put(key, node);
    currentBytes += newSize;
}

HashMap Load Factor Optimization

Problem: Default HashMap resizes multiple times as cache fills Solution: Pre-size to avoid resizing capacity / load_factor + 1 = initial HashMap size 1000 / 0.75 + 1 = 1334 → new HashMap<>(1334, 0.75f) Result: Zero rehashes, reduced GC pressure
Java 
// Bad — resizes HashMap multiple times as cache fills
Map<Integer, Node> map = new HashMap<>();

// Good — pre-size to avoid resizing
Map<Integer, Node> map = new HashMap<>(capacity * 4 / 3 + 1, 0.75f);

Deep Dive — Common Pitfalls

These are the bugs interviewers watch for. Getting the implementation right but falling into one of these traps is a common failure mode. Each pitfall includes a code example showing the bug and the fix.

LRU Pitfalls

🐛 Using ReadWriteLock Incorrectly
Java — BUG
// BUG: get() is NOT read-only — it modifies the linked list
lock.readLock().lock();
try {
    return cache.get(key); // MOVES node to head — this is a WRITE
} finally {
    lock.readLock().unlock();
}

Multiple threads holding read locks will concurrently modify the linked list, causing pointer corruption. Always use the write lock for both get() and put().

🐛 Forgetting to Remove from HashMap on Eviction
Java — BUG
Node lru = tail.prev;
removeNode(lru);
// map.remove(lru.key); ← MISSING! Dangling reference in HashMap

The HashMap now has a dangling reference to a disconnected node. This causes memory leaks and incorrect get() results (finds node in HashMap but it's not in the list).

🐛 Not Storing Key in Node

When evicting the tail node, you need its key to remove it from the HashMap. Without the key in the Node, you'd need an O(n) HashMap scan — breaking your O(1) guarantee.

LFU Pitfalls

🐛 Not Resetting minFrequency on New Insert
Java — BUG
Node newNode = new Node(key, value); // freq = 1
keyMap.put(key, newNode);
freqMap.computeIfAbsent(1, k -> new DoublyLinkedList()).addToHead(newNode);
// minFrequency = 1; ← MISSING! minFrequency might still be 5

Without resetting minFrequency = 1, the next eviction targets the wrong bucket (e.g., bucket 5 instead of 1), evicting the wrong entry or crashing on an empty list.

🐛 Not Incrementing minFrequency When Its Bucket Empties

When promoting a node from the minFrequency bucket and that bucket becomes empty, failing to increment minFrequency means the next eviction will try to evict from an empty list. Optionally, also remove the empty DLL from freqMap to prevent memory leak from accumulated empty lists.

General Pitfalls

🐛 put() on Existing Key Not Treated as Access

Both LRU and LFU must handle put(existingKey, newValue) as an access event: LRU moves it to head, LFU increments its frequency. A common bug is treating it as a simple value update without adjusting ordering/frequency metadata.

🐛 Capacity Zero Not Handled

If capacity = 0, every get() should return -1 and every put() should be a no-op. Without explicit handling, division by zero or negative capacity can cause unexpected behavior.

Deep Dive — Variants & Hybrid Approaches

LRU-K

LRU-K

Tracks the K-th most recent access time for each entry. Evicts the entry whose K-th access is oldest. LRU is LRU-1. LRU-2 requires an entry to be accessed at least twice before it's "established," which resists scan pollution. Commonly used in database buffer pools.

2Q (Two-Queue)

2Q (Two-Queue)

Uses two queues: A1 (FIFO) for new entries and Am (LRU) for entries promoted on second access. An entry must be accessed twice to earn LRU protection. Single-access scan items never leave A1, preventing scan pollution of the main LRU cache. Similar in spirit to LRU-2 but more practical to implement.

ARC (Adaptive Replacement Cache)

ARC (Adaptive Replacement Cache)

IBM's patented algorithm (expired 2024) that dynamically balances recency and frequency. Maintains two LRU lists — L1 (recency) and L2 (frequency) — plus "ghost" lists of recently evicted entries. The ghost lists detect which signal (recency or frequency) is more predictive of the current workload and dynamically adjust the size split. ARC is the gold standard for adaptive caching but was patent-encumbered until recently.

W-TinyLFU (Caffeine)

🏆 State of the Art — W-TinyLFU

Used by Java's Caffeine library, the best-in-class cache implementation. Achieves near-optimal hit rates across diverse workloads.

Count-Min Sketch

A probabilistic data structure that estimates the frequency of elements in a data stream using sublinear memory. It can overcount (due to hash collisions) but never undercount. W-TinyLFU uses it as a lightweight frequency estimator — far cheaper than maintaining exact counts per entry like classic LFU.

Window cache (1% of capacity, LRU): Absorbs burst traffic and new entries. Prevents cold start by giving new entries a chance to build frequency.

TinyLFU admission filter: A Count-Min Sketch estimates frequency with minimal memory. When an entry is evicted from the window cache, it must have higher estimated frequency than the entry it would replace in the main cache. This prevents cache pollution.

Main cache (99%, segmented LRU): Split into "probationary" and "protected" segments. Entries earn their way in through the frequency filter.

Periodic frequency decay: All counts in the Count-Min Sketch are halved periodically (e.g., every N accesses). This solves LFU's stale popularity problem — historical frequency naturally decays over time.

CLOCK

CLOCK Algorithm

An approximation of LRU used in OS page replacement. Entries are arranged in a circular buffer with a "reference bit." On access, the bit is set to 1. On eviction, a clock hand sweeps the buffer: if a bit is 1, clear it and move on; if 0, evict that entry. Approximates LRU without the overhead of maintaining a linked list — no pointer mutation on read, making it naturally concurrent-friendly.

Deep Dive — Interview Questions & Answers

High-Level Design Questions

Q1: Design a caching system for a web app serving 10K requests/second

Scale: At 10K RPS, an in-process LRU cache is sufficient (sub-microsecond latency). If data is shared across servers, add a distributed cache layer (Redis/Memcached) in front of the DB.

Policy: LRU for general web traffic — strong temporal locality. User sessions, auth tokens, and API responses all exhibit recency patterns.

TTL: Different TTLs by content type: static assets get long TTL (hours), API responses get short TTL (seconds to minutes).

Pattern: Cache-aside for simplicity. Application checks cache first, falls through to DB on miss, populates cache on response.

Thundering herd: When a popular key expires, 1,000 requests simultaneously try to populate it. Solve with singleflight/mutex per key — only one request fetches from DB, others wait for that result.

Thundering Herd — Double-Checked Locking Solution

The core pattern: check cache → miss → acquire lock → check cache again (someone else may have populated it while you waited) → still miss → you fetch. The second check is critical — without it, every thread still hits the DB, just serialized instead of concurrent.

Java — Global Lock (Simple)
public class ThunderingHerdCache<K, V> {
    private final LRUCache<K, V> cache;
    private final Object lock = new Object();

    public V get(K key, Function<K, V> dbFetcher) {
        // Check 1: no lock needed
        V cached = cache.get(key);
        if (cached != null) return cached;

        synchronized (lock) {
            // Check 2 (DOUBLE CHECK): another thread may have
            // populated this key while we waited for the lock
            cached = cache.get(key);
            if (cached != null) return cached;

            // Still miss — I'm the one who fetches
            V value = dbFetcher.apply(key);
            cache.put(key, value);
            return value;
        }
    }
}
// Downside: global lock — miss on "product:123" blocks
// threads waiting for unrelated "user:456"
Java — Per-Key Lock (Better Concurrency)
public class ThunderingHerdCache<K, V> {
    private final LRUCache<K, V> cache;
    private final HashMap<K, Object> keyLocks = new HashMap<>();

    private Object getLockForKey(K key) {
        // Short critical section — just get/create the lock object
        synchronized (keyLocks) {
            return keyLocks.computeIfAbsent(key, k -> new Object());
        }
    }

    public V get(K key, Function<K, V> dbFetcher) {
        // Check 1: no lock
        V cached = cache.get(key);
        if (cached != null) return cached;

        // Only threads requesting the SAME key compete here
        Object keyLock = getLockForKey(key);

        synchronized (keyLock) {
            // Check 2: double-check
            cached = cache.get(key);
            if (cached != null) return cached;

            V value = dbFetcher.apply(key);
            cache.put(key, value);
            return value;
        }
    }
}
// "product:123" and "user:456" now fetch in parallel
// Tradeoff: keyLocks map grows unboundedly — needs cleanup
🎯 Why Double-Check Matters

Without double-check: Every thread hits the DB — just serialized instead of concurrent. You turned a stampede into a queue, but still N queries.

With double-check: Thread 1 fetches and populates cache. Threads 2–1000 acquire the lock, see the value in cache on the second check, and return immediately. 1 DB query instead of 1,000.

Cache warming: On cold start, pre-populate from database for known hot keys to avoid a miss spike.

Q2: How would you design a CDN's edge cache eviction policy?

LFU over LRU: Popular content (viral videos, trending images) stays popular for days/weeks. LRU would evict them if a batch of new content is uploaded. LFU's frequency shield protects them.

Time-windowed frequency: Pure LFU has stale popularity — combine with windowed frequency counting (only count accesses in last 24h). W-TinyLFU's periodic decay achieves this naturally.

Size-weighted eviction: A 100MB video costs 20,000× more cache space than a 5KB CSS file. Consider frequency / size as the eviction priority — evict large, rarely-accessed items first.

Multi-tier: Memory (hot, small) → SSD (warm, large) → Origin (cold, fetch). Different eviction policies per tier.

Q3: You're designing a database buffer pool. Which eviction policy?

LRU-2 or 2Q, not simple LRU. Database workloads mix random lookups (high locality, cache-friendly) with sequential scans (full table scans — cache-hostile). A simple LRU cache would be destroyed by every SELECT * FROM large_table.

LRU-2: Requires two accesses before an entry earns cache residency. Single-pass scan pages never make it past the probationary stage.

Real-world: MySQL InnoDB uses a modified LRU with "young" and "old" sublists (similar to 2Q). PostgreSQL uses a CLOCK variant for shared buffers.

Q4: Your LRU cache's get() takes 2ms at p99 under high concurrency. Optimize it.

Diagnosis: The single global lock is contended. All threads compete for the same lock on every get().

Option 1 — Striped locking: Partition keys into N buckets, each with its own lock + LRU list. Reduces contention N-fold. Tradeoff: each partition has its own LRU order (less globally optimal eviction).

Option 2 — Caffeine-style buffered writes: Record access events in a lock-free ring buffer. A single drain thread periodically applies them to the LRU list. get() becomes nearly lock-free — just a HashMap read + buffer append.

Option 3 — CLOCK approximation: Replace the DLL with a circular buffer + reference bits. No move-to-head on access — just set a bit. Less accurate but eliminates list mutation on read entirely.

Q4: Design a cache for a Kubernetes controller that reconciles 500 resources every 5 minutes

Policy — LRU, not LFU: All resources are accessed at the same frequency (once per reconcile cycle). LFU degenerates to random eviction under uniform access. LRU naturally keeps the most recently reconciled resources cached.

Cache key: Resource identity — namespace/name/resourceVersion. The resourceVersion ensures stale versions don't pollute the cache.

Size-based limit: Resources vary in size (a ConfigMap might be 1KB, a large CRD might be 50KB). Use byte-based maxBytes tied to the pod's memory limit — not entry count. This directly maps to the pod's memory budget and prevents OOM kills.

TTL for correctness: Mutable resources need short TTL to detect upstream changes. Even with a watch stream, TTL acts as a safety net against missed events.

OnEvict callback: Fire a Prometheus metric on each eviction to track cache churn. Don't import Prometheus into the cache package — use a callback function to keep the cache generic.

OOM risk: An unbounded cache in a controller pod is a ticking time bomb. The pod will be OOM-killed under memory pressure. maxBytes is critical, not optional.

Q5: How would you handle cache warming after a pod restart?

Problem: Cold cache after restart = spike in cache misses = spike in backend/API server load. If multiple pods restart simultaneously (rolling deployment), the miss storm multiplies.

Implementation Questions

Q5: Modify your LRU cache to support per-entry TTL

Add an expiresAt field to the Node, set during put(), and check during get():

Java
static class Node {
    int key, value;
    long expiresAt;  // System.currentTimeMillis() + ttlMillis
    Node prev, next;
}

public int get(int key) {
    Node node = map.get(key);
    if (node == null) return -1;
    if (System.currentTimeMillis() > node.expiresAt) {
        // Expired — treat as cache miss, remove immediately
        removeNode(node);
        map.remove(key);
        return -1;
    }
    moveToHead(node);
    return node.value;
}

public void put(int key, int value, long ttlMillis) {
    Node newNode = new Node(key, value);
    newNode.expiresAt = System.currentTimeMillis() + ttlMillis;
    // ... rest of standard put() logic
}
⚠️ Common Follow-Up: "Expired entries still occupy memory"

Lazy expiry on get() ensures correctness (never serve stale) but doesn't reclaim memory from unaccessed expired entries. Two solutions:

• Background sweep thread — periodically scan and remove expired entries. O(n) cost per sweep. See TTL Integration section.

• DelayQueue<Node> — a priority queue sorted by expiresAt. A drain thread calls queue.take() which blocks until the next entry expires, then removes it. More efficient than periodic sweeps — zero wasted scans.

Q6: What happens if all entries in your LFU cache have the same frequency?

LFU degenerates to LRU within that frequency bucket. It evicts the least recently used entry among those with the minimum frequency. This is by design — the DLL within each frequency bucket maintains recency order, and eviction takes from the tail (least recent).

This is also why LFU is a poor choice for workloads with uniform access frequency — it provides no advantage over LRU but adds implementation complexity.

Q7: How does Java's LinkedHashMap implement LRU internally?

LinkedHashMap extends HashMap and threads a doubly-linked list through all entries. Each Entry has before and after pointers. With accessOrder=true, get() calls afterNodeAccess() which moves the entry to the tail (most recent). put() calls afterNodeInsertion() which checks removeEldestEntry() — if true, removes the head (LRU entry).

So internally, it's the exact same HashMap + DLL pattern — Java just provides it as a library class.
